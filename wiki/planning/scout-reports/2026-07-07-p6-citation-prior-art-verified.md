# Prior-art report: citation-intent classification, citation-accuracy checking, missing-literature detection, and retraction hygiene (HASS/archaeology)

## Executive summary

This is a **partially-solved problem with one clear gap and one near-total gap**. Citation-intent classification (does a citation support/contrast/background/use the cited work) is a mature research area with production services (scite.ai) and multiple open datasets/models (SciCite, ACL-ARC, MultiCite, ACT2) — but every one of them is trained and benchmarked on STEM/CS/biomedical corpora, and none address archaeology or HASS (Humanities, Arts, and Social Sciences). Citation-accuracy checking has split into two distinct sub-problems that the tooling landscape treats very differently: "does this reference exist / is its metadata correct" is well-served (RefChecker, CheckIfExist, multiple retraction-checkers, Crossref's now-free Retraction Watch data) — but "does the cited source actually semantically support the citing sentence's claim" is a 2025-26 research-frontier problem with exactly one close-fit open-source tool found (SemanticCite) plus several academic-paper-only implementations (CiteCheck, DeepSciVerify), none domain-adapted to HASS. Missing-literature/discourse-gap detection at the single-paper level — the user's stated core need — is the weakest area: everything found either operates at corpus/field scale (VOSviewer, CiteSpace), at researcher scale (Citation-Constellation), or is a general-purpose "find related papers" recommender (Connected Papers, Litmaps, ResearchRabbit, Inciteful) rather than a diagnostic tool that judges whether a specific paper's engagement with the literature is adequate. Retraction-hygiene checking is the most solved sub-problem of the five: Crossref now hosts Retraction Watch data for free via its standard API, and several thin wrapper tools already do exactly what's needed.

## Recommendations

**Use directly:**
- **Crossref REST API** (#21) for retraction-hygiene checking — no tool-building needed, just a per-DOI API call against the reference list already captured by the CEM (Claims-Evidence-Methods) extraction pipeline.

**Adapt approach:**
- **SemanticCite** (#8) — its four-class scheme (Supported / Partially Supported / Unsupported / Uncertain) with confidence score, reasoning, and evidence snippet is architecturally the right shape for "fair/accurate use of cited literature." Recommend studying its prompt design and retrieval pipeline (full-text retrieval → hybrid BM25+dense search → LLM judge) as a reference architecture rather than adopting the Streamlit/ChromaDB app wholesale — it has no confirmed licence, is single-maintainer, and has no HASS tuning.
- **RefChecker** (#7) — reference architecture for the "does this citation exist and is its metadata right" layer (useful as a pre-check before spending LLM budget on semantic support-checking); its multi-source fallback pattern (Semantic Scholar → OpenAlex → CrossRef → DBLP → ACL Anthology) is worth replicating, substituting in archaeology-relevant sources (e.g., Zenodo, JSTOR metadata, national archaeology repositories) where DBLP/ACL Anthology add nothing.
- **GraphMind / OpenNovelty** (#16 and companion paper) — the "extract contribution claims → retrieve comparable prior work → LLM-reason about what's covered vs. not" pipeline pattern is the right shape for missing-literature/discourse-gap detection, even though the implementation is CS/ML-specific and Semantic-Scholar-dependent.
- **MultiCite's** (#4) multi-sentence, multi-label citation-context scheme — the *idea* that a single citation context can carry multiple simultaneous intents is worth borrowing for a HASS-adapted labelling scheme, even though the CC BY-NC 2.0 licence and NLP-domain training data rule out reusing the model or data directly.

**Ignore:**
- **scite.ai** (#1) — proprietary, enterprise-priced API access, general-science training corpus with no stated archaeology/HASS coverage; not adoptable as a dependency and not worth building against.
- **ACT2** (#5), **allenai/scicite** (#3), **LITONG99/Citss** (#6) — all stale (2-4 years since last commit) and trained on CS/NLP or unverified STEM corpora; fine-tuning them for archaeological/HASS text would cost more than prompting a general LLM with a custom scheme.
- **Connected Papers / ResearchRabbit / Litmaps / Inciteful** (#17-20) — genuinely useful for a human doing literature discovery, but none perform the semantic "what does this paper ignore" judgement the user needs; they surface candidate papers, they don't audit engagement.
- **VOSviewer** (#26) and **Citation-Constellation** (#27) — wrong unit of analysis (field-level and researcher-level respectively, not single-paper).
- **Recite** (#13) — service currency could not be confirmed (site blocked automated access; last independent reviews are a decade old) and its function (format consistency) doesn't address the user's need regardless.

## Build-vs-adopt verdict

This has to be answered per sub-component, since the five categories the user asked about sit at very different points on the adopt-vs-build spectrum:

1. **Retraction/citation-hygiene (category 4): Adopt directly.** Add a Crossref `works/{DOI}` lookup to the existing extraction pipeline for every cited DOI the CEM graph captures, and check the `update-to`/relation fields for retraction notices. This is a few lines of code, not a project.

2. **Citation-accuracy / semantic-support checking (category 2, the "fair and accurate use" question): Build, informed by SemanticCite's architecture.** Nothing exists that is both (a) domain-adapted to HASS/archaeology and (b) integrated with a claims-evidence-methods graph. SemanticCite's classification scheme and retrieval-then-judge mechanism is the best available reference design (close enough that it should shape the schema — e.g., a `support_status` field with the same four values on each citation-context node), but the actual implementation should sit natively inside `extraction.json`/the research-assessor skill rather than wrapping an external unlicensed single-maintainer app.

3. **Citation-intent classification (category 1): Build with an LLM prompt, informed by SciCite/MultiCite/ACT2 label schemes.** Given this project already does structured LLM extraction (Claude-based, per the existing 8-pass workflow), the cheapest and most HASS-appropriate path is a custom classification prompt using an established scheme (SciCite's 3-class, or MultiCite's multi-label scheme) rather than fine-tuning or calling any of the existing SciBERT-era models, all of which are CS/NLP-domain and 2-4 years stale.

4. **Missing-literature / discourse-gap detection (categories 3 and 5, arguably the user's hardest and most novel ask): Build from scratch.** This is the one area that is genuinely unclaimed at the granularity the user needs (a single target paper, judged for which research discourses it engages vs. ignores). The closest patterns to borrow are GraphMind/OpenNovelty's claim-extraction-then-comparable-work-retrieval pipeline and RMC's "reviewer-flagged gap" framing, but both are CS/ML-specific, both depend on Semantic Scholar coverage that is known to be weak for archaeological grey literature and excavation reports, and neither attempts semantic discourse-community labelling (they recommend papers, they don't classify "processual vs. post-processual vs. indigenous archaeology" style engagement). Retrieval quality will likely be the binding constraint here — HASS/archaeology bibliographic coverage in OpenAlex/Semantic Scholar/Crossref is patchier than in CS/biomedicine, so any build should budget time for evaluating retrieval recall before investing in the LLM-judgement layer.

5. **Combine**: the realistic end-state is combining #1 (Crossref, off the shelf) + a custom LLM classifier informed by #2/#3's schemes + a custom retrieval-and-gap-reasoning layer informed by #4's patterns — there is no single existing tool or even pair of tools that covers more than one of the five requested categories at once, and none address HASS/archaeology at all.

**Search gaps/failures to flag honestly:** No Hugging Face Hub search tool was available in this session (only web search over huggingface.co), so HF model/dataset coverage for citation-intent classifiers may be incomplete — recommend a follow-up pass with direct Hub API access before finalising the design. Semantic Scholar's citation-count API was rate-limited (HTTP 429) during this session for the Citation-Constellation paper, so that specific number is unverified rather than fabricated. GitLab was not searched — low expected yield for this niche given GitHub's dominance in this space, but not confirmed empirically. No academic-literature search tool (e.g., a dedicated scholarly search MCP) was available; all paper-level findings came from WebSearch snippets rather than a structured citation database, so citation counts and precise author lists for most arXiv papers cited above were not independently verified beyond the one paper checked directly.
