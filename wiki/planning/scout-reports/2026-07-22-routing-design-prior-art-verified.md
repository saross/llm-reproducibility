## Executive summary

This is a partially-solved problem with one significant native-tooling finding that should
change the design before Phase 1 build starts. The skill/agent content-division question
(§7.1) is not solved by any external framework, but Claude Code itself ships a
harness-native mechanism — the `skills:` subagent frontmatter field — that already does
guaranteed, deterministic content injection at subagent spawn time, which is closer to what
the design is hand-rolling via orchestrator string-concatenation than anything found
elsewhere; this plausibly answers §7.5 (generated-agent-files) too, in the opposite
direction from what was anticipated (no build step needed, not "cheap tooling justifies a
build step"). The read-receipt question (§7.2) turned up one genuinely stronger pattern
(runtime-generated, cryptographically unforgeable "tool receipts") but it solves a
different problem — tool-call fidelity, not context-consumption — and adopting it would
require re-routing pushed content through a tool call, which the design deliberately did
not do; the version-line echo already in the design is, on current evidence, close to the
field's state of the art for the specific sub-problem it targets. The prompt-caching
economics question (§7.3) produced the most consequential and time-sensitive finding: the
project's own harness caps subagent prompt caching at a 5-minute TTL with a cold cache on
every spawn (confirmed in current Claude Code docs), and there is an open, unresolved
GitHub issue alleging subagent caching may not fire at all in some code paths — meaning the
"caching makes verbatim push negligible" assumption that holds for direct Anthropic API use
and for comparator eval harnesses (Inspect AI) may not transfer cleanly to this project's
Claude Code subagent architecture, and the Phase 3 cost gate should not assume a caching
discount without an empirical check on this harness. The corpus-screener question (§7.4)
is well supported: both open-source tooling (ASReview, paperfetcher, pyalex, habanero) and
two 2026 papers (Affinage; the LLMSurver corpus-filtration paper, peer-reviewed in
*Computers & Graphics*) converge on "deterministic API retrieval, narrow LLM/classifier
triage, human oversight" as the established shape for this task, supporting the design's
own instinct against a general `corpus-screener` agent.

## Recommendations

**Use directly:**
- pyalex and habanero (rows 22–23) for the deterministic-API layer of any `corpus-screener`
  implementation — mature, MIT-licensed, no reason to hand-roll CrossRef/OpenAlex clients.
- The `skills:` subagent frontmatter field (row 3) as the delivery mechanism for pushed
  instruments, replacing (or backstopping) the orchestrator's hand-rolled verbatim-inline
  step, provided the pre-flight check and read receipt are kept as a backstop against the
  documented silent-skip-on-missing-skill behaviour.

**Adapt approach (methodology sound, implementation doesn't fit):**
- The LLMSurver / corpus-filtration pattern (row 21) and Affinage's Stage-0 (row 20):
  adopt the *shape* — deterministic retrieval, then a narrow, scoped LLM/classifier triage
  step, human-supervised — for `corpus-screener`, without adopting either codebase (wrong
  domain/language for direct reuse).
- PydanticAI's static/dynamic instruction split (row 5) and promptfoo's regression-gate
  pattern (row 8): both validate design choices already made (the (a)/(b) split; the
  regression-gate maintenance rule) and could inform implementation details (e.g. using
  promptfoo's pass-rate-drop CI pattern to implement preregistration §8's regression test
  mechanically) without adopting either tool as a dependency.
- Tool Receipts (rows 9–10): the cryptographic-unforgeability *property* is worth keeping
  in mind as an escalation path if the version-line echo's compliance rate proves
  insufficient in practice — but only if pushed/pulled content is re-routed through an
  orchestrator-exposed tool call, which is a real architecture change, not a drop-in.

**Ignore:**
- Langfuse (row 7) and google/dotprompt (row 6): both solve "manage prompts as a
  product/team concern decoupled from code" or "compile templates for application code" —
  neither matches the frozen-instrument, git-history-is-the-audit-trail, solo-maintainer
  constraint, and both would add a dependency and a new drift surface rather than removing
  one.
- ASReview (row 18) and paperfetcher (row 19) as direct dependencies: right problem space,
  wrong mechanism (classical active learning, not LLM triage) or too stale, respectively.
  Their value here is corroborating evidence for the human-in-the-loop norm, not code to
  adopt.
- HELM and OpenAI evals (rows 16–17) as architecture models for §7.3: HELM avoids the
  question by not using LLM judges for most of its suite; OpenAI evals is stale and offers
  no new information beyond Inspect AI's more current and better-matched example.

## Build-vs-adopt verdict

1. **Skill/agent content division & push tooling (§7.1).** *Adapt* — not "adopt a
   framework," but adopt the harness's own `skills:` frontmatter mechanism (row 3) as the
   delivery layer for pushed instruments, keeping the orchestrator pre-flight check and
   read receipt as the backstop for its documented silent-skip behaviour. No external
   framework (LangGraph/CrewAI/AutoGen/PydanticAI) offers anything Claude-Code-specific
   here; their value is confirmatory (PydanticAI's explicit static/dynamic split validates
   the design's own (a)/(b) reasoning) rather than adoptable.
2. **Read receipts (§7.2).** *Build, informed by the approach found.* Nothing found is a
   drop-in upgrade to the version-line echo without an architecture change. The tool-receipt
   pattern (HMAC, runtime-generated, unforgeable) is strictly more robust than a self-report
   for the problem it solves (tool-call fidelity), but this design's problem (context
   consumption via inline push or file pull) doesn't have a tool call to sign under the
   current design. If the version-line echo's real-world compliance rate proves
   insufficient, the escalation path is to route pushed content through an
   orchestrator-exposed tool call and adopt receipt-style verification then — not now.
3. **Prompt-size economics (§7.3).** *Do not assume; measure on this harness before the
   Phase 3 cost gate.* The generic "caching makes push negligible" claim is well-supported
   for direct Anthropic API use (0.1× read cost) and matches what comparator eval harnesses
   (Inspect AI) do by default, but the project's own harness caps subagent caching at a
   5-minute TTL with a cold cache per spawn (row 13), and there is an open, unresolved issue
   suggesting subagent caching may not fire at all in some paths (row 14). Recommend: (a)
   re-check issue #29966's current state and the actual `cache_creation_input_tokens` /
   `cache_read_input_tokens` split on a real `fair-assessor` run before pricing the census;
   (b) if caching doesn't materialise, either pipeline invocations to preserve cache warmth
   within 5-minute windows or price the census at full uncached input rate; trimming the
   instrument to an "operational core" is not what any comparator does by default and
   should be a last resort, not the first lever.
4. **`corpus-screener` shape (§7.4).** *Combine* — the design's own instinct (deterministic
   API script + narrow LLM triage, not a general agent) is well supported by both tooling
   (pyalex, habanero — adopt directly) and two 2026 papers describing exactly this
   architecture (Affinage's Stage-0; the peer-reviewed LLMSurver pattern). Recommend
   confirming `corpus-screener` is demoted from "agent" to "script + narrow triage step" per
   the design document's own flagged suspicion.
5. **Generated-agent files (§7.5).** *Build was never necessary here* — the `skills:`
   frontmatter finding (row 3) most likely resolves this question in the opposite direction
   from what was anticipated: no build step is needed to keep agent-definition files short
   and single-sourced, because the harness already assembles subagent context from named
   skill files at spawn time. This is the strongest single finding of this scout pass and
   should be checked (ideally by a short spike: define one agent with a `skills:` pointer to
   an existing canonical instrument file and confirm the content actually lands in the
   subagent's context) before any build-script work is considered.
